## Repository Structure

```
Claude_Experiments/
├── CLAUDE.md          ← this file; update it as the repo evolves
└── <experiment-dirs>  ← one directory per experiment or project
```

As experiments are added, each should live in its own directory with at minimum:
- A short `README.md` or inline comments explaining what it demonstrates
- Its own dependency/config files (e.g. `package.json`, `requirements.txt`)
- A clear entry point (e.g. `main.py`, `index.ts`, `run.sh`)

## Development Conventions

### Branching
- Default branch: `main`
- Feature/experiment branches: `<topic>/<short-description>` (e.g. `experiments/tool-use-demo`)
- Claude Code automated branches: `claude/<task-slug>` (auto-generated by the harness)
- Never push directly to `main` without review when collaborating.

### Commits
- Use short, imperative subject lines: `Add streaming demo`, `Fix token count off-by-one`
- No "WIP" commits on `main`; squash or rebase before merging
- Each commit should leave the repo in a working state

### Pull Requests
- PR title mirrors the commit convention (imperative, concise)
- Body: one-paragraph summary + bullet test plan
- Keep PRs small and focused; one experiment or fix per PR

## AI Assistant Instructions

These rules apply to Claude Code and any other AI assistant working in this repo.

### General
- Read `CLAUDE.md` at the start of every session; treat it as ground truth
- Update `CLAUDE.md` when you add a new experiment, change conventions, or discover
  something that would have saved you time
- Prefer editing existing files over creating new ones
- Do not add comments that explain *what* the code does — only the *why* when non-obvious
- Do not add unused error handling, feature flags, or backward-compat shims

### File Creation
- New experiments → new subdirectory, not files in the root
- Never create `*.md` documentation beyond `README.md` and `CLAUDE.md` unless asked
- Keep config files (`.env`, secrets) out of git; add them to `.gitignore`

### Dependency Management
- Pin dependency versions in lock files when a package manager is used
- Do not upgrade transitive dependencies speculatively

### Security
- Never commit credentials, API keys, or tokens — use environment variables
- Validate inputs at system boundaries; trust internal code
- Flag any OWASP Top-10 risks immediately rather than deferring

### Testing
- If an experiment has runnable code, verify it actually runs before committing
- Unit tests are encouraged but not mandatory for quick demos
- Integration tests that hit live APIs should be clearly marked and skippable offline

## Claude API / Model Conventions

When writing code that calls the Anthropic API or uses Claude models, follow these defaults:

- **Default model:** `claude-sonnet-4-6` (balanced capability and cost)
- **High-stakes / complex tasks:** `claude-opus-4-8`
- **Fast / cheap / high-volume:** `claude-haiku-4-5-20251001`
- Use the `anthropic` SDK (Python) or `@anthropic-ai/sdk` (TypeScript/Node)
- Always handle streaming errors and connection resets gracefully
- Add `max_tokens` explicitly — never rely on the default
- Cache repeated system prompts with prompt caching when applicable
